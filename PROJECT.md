# PROJECT.md — key-keeper

Локальный менеджер лицензий и ключей. Полное техническое задание.

---

## 1. Видение

Локальное offline-приложение для хранения лицензионных ключей приложений, программ
и подписок с контролем сроков их действия. Все данные шифруются мастер-паролем и
хранятся только на устройстве пользователя — без сервера и без обращений в интернет.

**Главная ценность:** в один взгляд видеть, что активно, что истекает в ближайшие дни
и что уже просрочено.

**Архитектура развития:** сначала web-приложение (React), затем десктоп-обёртка Tauri
поверх того же кода. Слои хранения и шифрования держим абстрактными, чтобы переход на
десктоп не требовал переписывания логики.

---

## 2. Цели и принципы

- **Безопасность данных** — ключи нигде не хранятся в открытом виде.
- **Offline-first** — никаких сетевых запросов в рантайме (требование доверия).
- **Минимум кликов** — ключевые действия (добавить, скопировать, показать) в один тап.
- **Apple-стиль** — минимализм, чистота, много воздуха, спокойная палитра.
- **Защита от потери** — экспорт-бэкап, напоминания, мягкое удаление.

## 3. Технологический стек

| Назначение | Технология |
|---|---|
| UI-фреймворк | React + Vite + TypeScript |
| Стили | Tailwind CSS (darkMode: 'class', светлая тема по умолчанию) |
| Состояние | zustand |
| Хранилище | IndexedDB через Dexie.js (вынести в отдельный модуль `storage`) |
| Шифрование | Web Crypto API (AES-GCM; ключ из мастер-пароля через PBKDF2) |
| Excel-экспорт | SheetJS (xlsx) |
| Даты | dayjs |
| Иконки | lucide-react |
| Десктоп (этап 12) | Tauri 2.x |

Tailwind компилируется штатно через Vite — классы `dark:` работают (в отличие от
артефактов без компилятора).

## 4. Модель данных

### License (запись о лицензии)

| Поле | Тип | Описание |
|---|---|---|
| `id` | string | Уникальный идентификатор |
| `name` | string | Название софта / подписки |
| `licenseKey` | string | Ключ лицензии (в UI маска `••••`) |
| `accountLogin` | string | Логин / имя пользователя, выданное вместе с ключом (необяз., может быть пустым) |
| `platform` | enum | `windows` \| `ios` \| `android` \| `web` (Windows / iPhone / Android / Веб-сервис) |
| `category` | string \| null | Ссылка на справочник категорий, может быть пустой |
| `activationUrl` | string | Ссылка на страницу активации (откр. в новой вкладке) |
| `purchaseUrl` | string | Где куплено — ссылка или текст |
| `linkedEmail` | string | Почта привязки (необяз.) |
| `purchaseDate` | date \| null | Дата покупки (опц.) |
| `expiryDate` | date \| null | Дата окончания |
| `isPerpetual` | boolean | Бессрочная лицензия (взаимоисключает expiryDate) |
| `status` | enum | Вычисляется: `active` \| `expiring` \| `expired` \| `perpetual` \| `archived` |
| `remind` | boolean | `false` = не актуально, не напоминать |
| `comment` | string | Произвольный комментарий |
| `images` | blob[] | Массив фото в формате WebP (галерея) |
| `tags` | string[] | Теги (опц.) |
| `createdAt` | datetime | Дата создания |
| `updatedAt` | datetime | Дата изменения |

### Category (справочник категорий)

| Поле | Тип | Описание |
|---|---|---|
| `id` | string | Идентификатор |
| `name` | string | Название категории |

**CRUD категорий:** создать, переименовать, удалить.
**При удалении категории** привязанные ключи НЕ удаляются — у них поле `category`
очищается (становятся «без категории»).

### Служебное

- `schemaVersion` — версия схемы данных, хранится в данных и в файле бэкапа.
  Нужна для миграций будущих версий и совместимости старых экспортов.

## 5. Функциональные требования

### 5.1 Работа с ключом и логином в интерфейсе

**Ключ (`licenseKey`):**

- По умолчанию замаскирован (`••••`).
- В форме редактирования: иконка-глаз (показать / скрыть), иконка-копировать.
- На карточке списка: клик по полю маски → копирование в буфер; зелёная обводка и галочка «Ключ скопирован» на ~2 сек.
- После копирования буфер автоматически очищается через **120 секунд**.

**Логин (`accountLogin`):**

- Необязательное поле: у части лицензий вместе с ключом выдаётся логин / имя пользователя.
- Если поле пустое — на карточке не показывается (в форме поле ввода доступно).
- Если заполнено — **отображается открытым текстом** (без маски):
  - в форме: обычное поле + копировать;
  - на карточке: отдельная строка под ключом, клик → копирование, зелёная обводка и «Логин скопирован».
- В поиске и Excel-экспорте (Этап 6 / 10) — учитывать как отдельное поле.

### 5.2 CRUD записей

- Форма добавления / редактирования со всеми полями модели.
- Подтверждение при удалении.
- Мягкое удаление через архив (вместо немедленного жёсткого удаления).
- Предупреждение о дубликате, если `licenseKey` уже существует.

### 5.3 Фото

- Несколько фото на одну запись (галерея).
- При добавлении каждого: изображение → canvas → даунскейл до макс. **1600px**
  по большей стороне → `canvas.toBlob('image/webp', 0.92)` → сохранение blob.
- Блобы шифруются вместе с записью.
- В карточке — превью (первое фото + счётчик); полная галерея по клику;
  удаление отдельных фото.

### 5.4 Список и фильтрация

- Поиск по названию / ключу / логину (`accountLogin`) / комментарию — **in-memory** по массиву zustand.
- **Минимум 3 символа** для срабатывания фильтра; при 0–2 символах показываются все записи.
- **Учёт раскладки клавиатуры:** запрос на русской раскладке находит латинские названия
  (и наоборот). Пример: забыли переключить раскладку и вводят `ьшс` — находится
  `Microsoft`. Маппинг по физическим клавишам ЙЦУКЕН ↔ QWERTY.
- При активном поиске в списке остаются только подходящие карточки; совпадения
  **подсвечиваются** в тексте (название, комментарий; ключ и логин — если совпадение по
  `licenseKey` / `accountLogin`, индикатор без раскрытия значения).
- Фильтры: платформа, категория, статус, тег.
- Сортировки: по сроку окончания, названию, платформе, дате добавления.
- Массовые действия: выделить несколько → архив / удаление.

### 5.5 Логика сроков

- «Дней до окончания» считается от **локальной полуночи** (не от времени с секундами),
  чтобы «осталось N дней» было корректно весь день.
- `expiring` = осталось ≤ 14 дней (порог вынести в настройки).
- Цветовая индикация статусов:
  - 🟢 `active` — активна
  - 🟡 `expiring` — истекает скоро
  - 🔴 `expired` — просрочена
  - ⚪ `perpetual` — бессрочная
  - приглушённо `archived` — архив / не актуально

### 5.6 Дашборд

- Сводка: количество active / expiring / expired.
- Блок «истекает скоро».
- Счётчик истекающих в заголовке вкладки браузера (напр. `(3) key-keeper`).

### 5.7 Напоминания

- Учёт флага `remind` (если `false` — запись не попадает в напоминания).
- Опционально: Notification API при открытии приложения для истекающих ключей.

### 5.8 Экспорт и импорт

- **Зашифрованный бэкап `.vault`** — основной способ хранения и переноса.
  Содержит `schemaVersion`. Импорт восстанавливает все записи.
- **Excel-экспорт** — стильный отчёт: шапка, заливка по статусам, авто-ширина колонок.
  - **ВНИМАНИЕ:** Excel НЕ шифруется, данные в открытом виде.
  - Обязателен тумблер «маскировать ключи» (выгружать `••••` вместо ключа).
  - Предупреждение перед выгрузкой о том, что файл содержит данные открытым текстом.

### 5.9 История изменений

- Audit log по записи: что и когда менялось (этап 11).

## 6. Требования безопасности

- Вход по мастер-паролю; из пароля через PBKDF2 формируется ключ AES-GCM.
- При первом запуске — **явное предупреждение**, что восстановление пароля невозможно
  (забыл = данные потеряны навсегда), и настойчивое предложение делать бэкап.
- В IndexedDB хранится **только зашифрованный блоб** (включая фото). Открытых ключей
  нет нигде.
- Авто-блокировка по бездействию — по умолчанию **60 минут** (настраивается).
- Тумблер «держать сессию до закрытия приложения» — ключ живёт в памяти, на диск не
  пишется.
- «Постоянно залогинен» — **только в десктоп-версии** (этап 12) через keychain ОС.
  В web этого пункта НЕТ: ключ на диск в открытом виде не сохранять никогда.
- Никакого вывода ключей и паролей в консоль или логи.

## 10. Версионирование

Два независимых номера версий:

| Версия | Где хранится | Назначение |
|---|---|---|
| **Версия приложения** | `package.json`, `src/constants/app-info.ts` | SemVer для релизов UI/логики (`MAJOR.MINOR.PATCH`) |
| **schemaVersion** | внутри vault-блоба, файл `.vault` | Миграции структуры данных |

### Версия приложения (SemVer)

- Формат: `MAJOR.MINOR.PATCH` (сейчас фаза `0.x.y` — до первого стабильного релиза).
- **MINOR** — по завершении значимого этапа из `PLAN.md` (напр. `0.1.0` = Этап 1).
- **PATCH** — исправления и мелкие доработки внутри этапа (`0.3.1` = дополнение этапа 3, логин).
- **MAJOR** — `1.0.0`: первый стабильный релиз; далее — несовместимые изменения UX/API.
- При bump версии обновлять **оба** файла: `package.json` и `src/constants/app-info.ts`.
- В `app-info.ts` также фиксировать `APP_RELEASED_AT` (дата и время выпуска версии, ISO 8601).
- Подвал приложения: `key-keeper vX.Y.Z · Леонов · ДД.ММ.ГГГГ, ЧЧ:ММ`.

### schemaVersion (данные)

- Целое число, начиная с `1`. Увеличивается только при изменении структуры vault.
- Не связана с версией приложения.

## 11. Ограничения и правила среды

- Все файлы и папки — только в `F:\Projects\Cursor\Work\key-keeper`. Диски C: и D: не трогать.
- PowerShell 5.x: без `&&`, команды построчно; `cd F:\path` (не `cd /d`).
- Абсолютные пути в командах создания; перед созданием проверять `Get-Location`.
- Имена файлов/папок: латиница, нижний регистр, дефис. Без пробелов и кириллицы.
- Комментарии в коде — русские (зачем, не что). Имена переменных — английские, понятные.
- Секреты только в `.env` через `process.env.X`; `.env` всегда в `.gitignore`.
